奔跑吧ansible（上）

发表于 2018-11-21

概述

ansible特点：

  • 在所有主机之间并行执行
  • 执行下一个task之前会等待上一个task在所有主机中执行完毕
  • ansible会按照确定的顺序执行
  • 幂等性：保证结果如你所期望（已经是了，就不做任何事）
  • 简单抽象，也就是没有一个统一的接口抽象。比如在centos下装包用yum，ubuntu下面用apt。需要在编写playbook时为不同OS执行不同操作
  • 复用（模块复用、role层面的playbook复用）Ansible Galaxy在线role仓库

playbook的本质就是一个字典组成的列表。确切地说，一个playbook就是一组play组成的列表

每个play必须包含的：

  • task:需要在这些主机上执行的任务

task

一个play可以有多个task，task有可选的name，可以使用–start-at-task 来从中间的task开始执行playbook。这里需要用name来引用task

每个task必须包含由模块名字组成的key，以及由传到模块的参数组成的value。如果直接在模块后面接参数，比如“apt: name=nginx update_cache=yes”,其参数只有一个，就是一个“name=nginx update_cache=yes”的字符串。指定多个参数，徐要用到字典。

执行ansible后发送变化了么？主机状态跟踪

对应模块返回字段changed,True则表示被改变了，执行playbook会打印黄色的提示。task执行时检查节点状态是否改变，如果已经和模块参数匹配就直接返回ok。这个changed标志可以配合handler来触发额外的动作。

关于变量

什么时候必须使用引号，

对于这种形式，YAML解释器会认为是一个内联字典,比返回一个错误。这种情况下就需要把参数””起来，（” -a foo”）。类似的原因，参数中有”,”也需要引号包裹。

特别地：debug模块的msg参数需要引号来捕捉空格。所以需要用到单引号与双引号。

对于旧版本：$var来引用变量值

handler

ansible识别到task改变了系统状态就会触发通知机制，task将handler的名字作为参数传递，以此来通知handler

关于handler：ansible只会在所有任务执行完后执行handler，而且即使notified多次也只会执行一次。handler按照play中定义的顺序执行，而不是notified的顺序。

陷阱：

  1. 运行ansible
  2. 一个task改变了状态并引发notify
  3. 后面的task执行错误，ansible退出
  4. 修复了playbook
  5. 再次运行playbook
  6. 重新开始执行并没有task状态汇报，所以ansible并不会执行handler

目的

写一个ansible的自定义的module，为达到练习的目的，要求如下：

  1. 用一个action参数来控制行为
  2. 不同的action取值要有不同的参数
  3. 对于异常需要返回错误
  4. 写出详细注释

计划

  • action=print_hello,则直接返回hello,不需要参数（模拟不带参数）
  • action=send_ones_respects,带个name参数，返回为谁问安，如果name=’Brigitte’,则从错误流返回，若是name=’Sombra’,则是失败执行（分别模拟带参数、错误返回、失败返回）

命令用途

host aggregate是在availablity zone之后出现的一个功能，让计算节点的分组更加灵活。在az中常做得是按照机房划分，机架等。在host aggregate中可以按照硬件性能划分计算节点，除此之外还可以按照给谁使用划分。对于第一种常见的场景就是划分出大内存组、高频CPU之类的；对于第二种我们可以把指定超配比比较高的用于开发环境，把稳定的低超配给用户使用。

host aggregate生效的途径是通过metadata配合filter来实现的，有两种方式来使用aggregate这个特性。

  1. 先给aggregate设定metadata（nova aggregate-set-metadata aggname key=value）然后给一个flavor设定同样的key=value（nova flavor flavorname set key=value）来实现aggregate和flavor关联，最后在nova_scheduler节点的nova.conf中配置AggregateInstanceExtraSpecsFilter即可
  2. 为aggregate指定一个az（这个az和这个host aggregate是一一对应的），为这个aggregate设置metadata比如（nova aggregate-set-metadata aggname cpu_allocation_ratio=5，再在nova_scheduler添加AggregateCoreFilter、AggregateRamFilter，AggregateDiskFilter。这样nova-scheduler会根据当前host Aggregate中metadata的ratio值去找合适的计算节点，如果没有找到就会使用全局的ratio
