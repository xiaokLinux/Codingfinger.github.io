命令用途

host aggregate是在availablity zone之后出现的一个功能，让计算节点的分组更加灵活。在az中常做得是按照机房划分，机架等。在host aggregate中可以按照硬件性能划分计算节点，除此之外还可以按照给谁使用划分。对于第一种常见的场景就是划分出大内存组、高频CPU之类的；对于第二种我们可以把指定超配比比较高的用于开发环境，把稳定的低超配给用户使用。

host aggregate生效的途径是通过metadata配合filter来实现的，有两种方式来使用aggregate这个特性。

  1. 先给aggregate设定metadata（nova aggregate-set-metadata aggname key=value）然后给一个flavor设定同样的key=value（nova flavor flavorname set key=value）来实现aggregate和flavor关联，最后在nova_scheduler节点的nova.conf中配置AggregateInstanceExtraSpecsFilter即可
  2. 为aggregate指定一个az（这个az和这个host aggregate是一一对应的），为这个aggregate设置metadata比如（nova aggregate-set-metadata aggname cpu_allocation_ratio=5，再在nova_scheduler添加AggregateCoreFilter、AggregateRamFilter，AggregateDiskFilter。这样nova-scheduler会根据当前host Aggregate中metadata的ratio值去找合适的计算节点，如果没有找到就会使用全局的ratio

nova的版本化(VersionedObject)

发表于 2018-07-09

接上一篇nova的面向对象，上一篇有提到objects化的核心就是实现数据和数据的操作绑定在一起，那么nova是如何实现的呢？

回顾

回顾一下上一篇提到的几个关键点(对于这里的数据类型是关键点)：
一、限制一下python的无需声明数据类型的特性，让数据类型限制一下

  1. 为了减少数据库类型的bug
  2. 提高数据库模型的稳定性，可搭配任意的关系型数据库

二、每个对象自己带有to_primitive()和from_primitive()方法,这种方式使得序(反)列化的时候具有上下文敏感性

  1. to_primitive : 进行序列化
  2. from_primitive : 反序列化

三、对象在更新的时候，实现差量更新

  1. 有时候对象需要保存在数据库的值并没有改变，就不需要去尝试连接数据库进行任何的读写操作
  2. 更新精确到字段，不需要整条记录都更新(这个感觉上开销是差不多的,还没验证)

数据类型

对于第一点，nova定义了和实现了自己的字段类型。pike版本这部分代码在oslo_versionedobjects.fileds模块下该模块完整代码链接

nova对用到的数据类型划分为两种，

  1. 严格定义的数据类型，比如各种String，int，float相关的。这类数据类型会对应到底层的数据类型
  2. 半严格的数据类型，AutoTypedField。这类数据类型没有绑定具体的数据类型可以对应int也可以对应其他类型

严格的数据类型体系

这个oslo_versionedobjects.fileds模块定义了一个抽象字段类型AbstractFieldType，其中定义了抽象方法

  • def coerce(self, obj, attr, value): ：将VersionedObject（版本化）对象的属性的值强制类型转换为需要的类型
  • def from_primitive(self, obj, attr, value): ：将将VersionedObject对象的属性的值进行反序列化
  • def to_primitive(self, obj, attr, value): ：序列化一个将VersionedObject对象的属性的值
  • def describe(self): ：返回字段的描述
  • def stringify(self, value):：返回一个字段表示版本的短字符串

接着对抽象类AbstractFieldType提供一个默认的实现FieldType类，通过源码可以看到这个类对coerce、from/to_primitive等抽象方法提供了一个默认的实现并定义为类方法，具体细节是

  • coerce ： 直接返回值，不进行任何强制转化
  • from_primitive ：默认直接返回value
  • to_primitive ： 默认直接返回value
  • describe : 默认返回这个版本化对象的类名
  • stringify ：默认返回传入参数的str(value)

nova的面相对象模型是如何抽象的

发表于 2018-07-06

蓝图 :
nova object化，挖坟wiki
蓝图发布时间 : 2013-2014

概述

nova中有一个模块是object，里面定义了一个base模块，base模块中NovaObjectRegistry，NovaObject，ObjectListBase，NovaObjectSerializer几个类。

这个nova.object就是nova的object化，这是Dan Smith推动下在2013年完成的。object化主要是为了实现四个功能：

  1. 解耦合.
  2. 申明Object 属性的类型.
  3. 减少写入DB的数据量.
  4. 实现dB API

nova的架构为了安全性隔离，nova-compute不直接操作数据库而是通过消息队列远程调用nova-conductor去完成数据库操作。在远程过程调用过程中nova-compute会把obj-method传给conductor。根据nova架构nova-compute和nova-conductor是可以分开部署，而且可以是不同版本的。object化之后，nova-conductor根据消息队列中的对象名称，实例化一个即可，这个对象无论nova-compute和nova-conductor版本是否一致都是继承于同一个基类，这样就实现了版本兼容。具体细节，以一次调用为例

  1. nova-compute想远程调用a.xx方法
  2. nova-compute通过to_primitive把a serialize 化,转成一个叫primitive的dict
  3. Conduct从消息队列获取到字符串，转成dict比对版本号，决定实例化那些属性
  4. Conduct通过from_primitive 把a 实例化
  5. Conductor的Object和db的model是一致的(identical), 这样就实现了Compute node 上的Object和db的松耦合. 可实例化的Object类型是通过metaclass 来控制的.

为例减少数据库数据关于类型的bug, Object的Fields就需要明确声明自己的类型.

Object有一个_change_field 属性, 用来记录变化的量, 从而减少写入dB(save())和回写到compute node的数据量(get_instance_by_uuid).

将各自的数据库调用封装在内部，以前所有的代码调用库方法都是先去db.api写需要的xxx方法，然后在代码中调用。随着功能越复杂db.api将越来越复杂而且难以管理。object化之后将各自操的数据库操作封装在各自的内部

蓝图内容

目的

无论是直接操作数据模型还是通过rpc都采用一个方式操作数据库。把数据和数据操作绑定在一起，特别是序列化和实际的数据库实现。代码隔离具体的数据库实现，以便实现滚动升级。摒弃直接调用db.api的方式，在具体的操作数据库类中执行属性检查，以便save()实现可以解耦

理由

原来的在nova中，在通过rpc调用之前所有的内容都会被转化为“原语”，传递原语意味着我们使用非OO接口来对这些项进行更改。

原函数：数据库API中提取的SQLAllchemy对象的直接表示。这种转换是不好的，因为当数据库模型变了，则原函数也需要改变

实现接口

Object Class Registry

这个接口是为了防止客户端随机实例化远程端点的任何对象，将维护一个注册表来保留我们需要实例化的对象。每个对象都将包含一个主要/次要的版本号。注册表通过元编程来实现
注意：这不是实例注册表，它仅用于托管可以实例化的类。它不会尝试跟踪其他服务正在运行或者正在使用的服务

Strictish Object Properties

定义一些属性的数据类型，在nova中存在很多数据类型的bug，比如很多数据类型在传递时是字符串在MySQL中却是int型。这种不统一的数据类型导致了同样的代码切换后端数据库就不可用了。这里的目标就是定义哪些对象应该严格定义，以减少数据类型引起的bug

Inbuilt Serialization

现在的jsonutils.to_primitive()能够序列化任何对象，这是不好的。现在让每个对象自己带有to_primitive()和from_primitive()方法,这种方式使得序(反)列化的时候具有上下文敏感性。为了让这种方式生效并且对使用者透明，oslo_rpcproxy和rpcdispatcher类将会调用例程的to_primitive和from_primitive方法。
我们可以使用isinstance(foo, ThisMagicObject)来判定是否是预期实例来避免中断

Object Implementation

nova因为需要隔离数据库驱动程序，所以有一个子类化的对象提供默认的实现来替代实例本身操作数据库(SQLAlchemy) <这里到底是说ORM的model还是说在ORM上封装数据操作层，猜测是后者等待阅读源码验证一下>
每个object类都提供了查询创建接口实现。对属性的更改会自动跟踪，使用save操作可以轻松地执行选定属性更新(类似于今天使用的instance_update()方式)

Magic Remoting Behavior

对象的每个方法(无论是类方法还是普通方法)都可以加上一个辅助(helper)修饰器,这个修饰器会提供自动的远程调用。这个修饰器是否生效取决于组件是否应该直接访问数据库，比如nova-api本来就可以直接操作数据库，所以修饰器会调用实际的对象实现来操作数据库。再比如nova-compute这个本身是不能访问数据库的，那么修饰器会拦截访问并尝试调用修饰器的方法来把数据层操作通过rpc导向nova-conductor

并发相关概念

什么是并发

如果两个事件互不影响，谁先执行谁后执行都没关系（顺序无关）那这两个事件就是能并发的。比如一般情况下去地铁站和吃冰棍并发，因为你既可以现吃冰棍再去地铁站，也可以先去地铁站再吃冰棍，甚至一起都可以，只要你高兴就好。这种事件就满足互不影响的条件，所以称吃冰棍和去地铁站互为并发事件。

为什么需要并发

传统处理问题的方式是执行一个序列，一步步做。这种模式不适合处理大规模的问题，因为存在以下的限制

  1. 处理作业的能力受运行该程序的机器(单机，CPU内核)性能限制
  2. 在进行IO处理时，CPU虽然空闲，但是程序在等待IO结果这进一步地限制处理作业的能力

在并发的情况下解决方式

  • 把作业分散到可并行处理工作的多个处理单元上
  • 在IO等待的时候保持响应，即使IO还没完成

对于问题1肯定需要并行处理器解决方案，实际情况也是使用多线程和多处理模型来解决
对于问题2需要看解决的问题细节，web服务需要为多用户提供服务不能因为某一个请求的IO就停滞这就需要并发

内存超配比、启动失败

概要

时间 ：2018年07月04日13:58:41
事件 ：计算节点有三台虚拟机，其中一台挂掉。重启时失败，提示内存不足。但是请求资源是在超配比范围内的
详细描述 ：free查看物理机220G内存，还有87G free，内存超配比为2.0。启动失败的虚拟机要求的内存是96G。按照openstack的过滤器，220*2 - （220-87）> 96。是不应该出现内存不足才对

TroubleShoot
  1. 查看nova-compute日志，发现已经生成xml了，xml中配额是96G。用这个xml去启虚拟机出错，报错是libvirt
  2. 尝试去找libvirt日志，发现没有开。去message下查找libvirt的日志。发现是因为物理机内存限制
  3. 在物理机查看交换分区大小，发现交换分区才4G。远小于96-87的差额。
  4. 完成TroubleShoot
总结
  1. 内存超配比对于差额的容忍度是与交换分区大小有关的
  2. 重启虚拟机是不走调度的（我也不知道我为啥一想到超配比就去查scheduler日志）
  3. 虚拟机在超配比的限制下全部运行时不一定会顺风顺水，当虚拟机实际使用内存总和大于物理机内存+物理机虚拟内存时就会翻车

oslo_vif

这个组件的作用？

用来在OpenStack中插入和拔出虚拟接口（VIF）的库。它提供：

  1. 表示各种虚拟接口及其组件的版本化对象
  2. 提供plug()和unplug()接口的基本VIF插件类
  3. 两个网络后端插件OpenVswitch和Linux-bridge

os-vif旨在定义用于在OpenStack中表示VIF类型的通用模型。 除了包含的两个插件外，其他网络后端的所有插件都保存在单独的代码存储库中。

oslo_content

这个组件的作用？

用来保存请求的上下文信息的基类，有两个作用：

  1. 保存web请求的安全信息
  2. 用来为oslo.log提供上下文详情

RequestContext基类的作用，提供了那些方法？

这个类的每个实例化对象都保存了一个web请求上下文信息。该类中保存的字段有：请求的token信息，用户信息，项目信息，服务信息等；在实例化时会为每个对象生成一个request_id在项目中唯一表示该对象。还提供了一系列其他方法以方便OpenStack其他项目调用。主要有：

  1. global_id（）返回全局id，如果我们想使用一个全局的request_id可以使用此方法
  2. from_dict（values，**kwargs）使用给定的对象创建上下文对象
  3. from_environ(environ, **kwargs)：使用环境变量创建上下文对象。
  4. to_dict()：返回一个包含上下文属性的字典。
  5. to_policy_values()：为oslo.policy组件提供上下文信息。
  6. update_store()：在当前线程中保存该上下文对象
  7. get_logging_values()：为oslo.log组件生成日志信息提供上下文信息。

如何使用RequestContext类?

一般用法是创建一个继承自RequestContext类的子类，并在子类中封装项目自身所需的其他上下文属性和方法。例如nova的nova.context.RequestContext添加了db_connection、mq_conection、timestamp等属性，也添加了get_auth_plugin()、elevated()、can()等方法。

此外，nova在nova.context.RequestContext的基础上继续封装了get_context()、get_admin_context()，判断用户角色的方法is_user_context()等可以更加方便在Nova项目中使用上下文信息。定义在nova.context.py中

nova

管理只是提供管理功能和调用接口，你想加快硬盘就去找硬盘管理官讨要一块硬盘，想换一个操作系统就去找操作系统管理员。但是管理的资源来自何处，所以对应的还有对资源提供者的抽象（resource provider），管理员需要频繁地和提供者确认资源使用情况
